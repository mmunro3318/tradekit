# Machine Learning Applications in Intraday Crypto Trading

## Overview

This report evaluates meta-labeling as a trade filter, probability calibration methods, model family selection (gradient boosting vs. logistic regression), concept drift detection and retraining governance, and explainability techniques — closing with a direct, evidence-based answer to whether ML reliably beats simple rule-based systems in trading specifically. The consistent theme: meta-labeling is the single most validated, well-specified ML application in this domain, but the broader academic evidence on ML's ability to beat simple rules in live trading is far more sobering than the retail-strategy literature often suggests.

## 1. Meta-labeling architecture

Meta-labeling is a two-layer system in which a primary model (which can be as simple as a rule-based signal generator, not necessarily ML itself) proposes trade signals, and a secondary "meta-model" predicts whether each individual signal should be taken or skipped, acting purely as a filter rather than a direction predictor. The architecture's core value proposition, stated consistently across sources: "meta-labeling will increase your F1-score by filtering out false positives, where the majority of positives have already been identified by the primary model" — the secondary model's job is exclusively to determine whether a positive from the primary model is true or false, not to discover new opportunities the primary model missed. One source frames this succinctly as "conditional trust": rather than demanding a single model produce near-perfect predictions, the system embraces a layered division of labor where the primary model identifies opportunities and the meta-model evaluates conditions under which those opportunities are actually worth taking.[^1][^2][^3][^4][^5]

The standard implementation workflow, consistent across sources, closely mirrors the questionnaire's own E.207–208 design: train the primary model or rule first, then define ground-truth labels via the triple-barrier method (upper barrier = profit target, lower barrier = stop-loss, vertical barrier = time limit), then train a secondary classifier on contextual features — typically volatility, order flow metrics, spread/depth, macro features, and signal strength — to predict whether that specific signal instance would have hit the profit barrier or the loss barrier. A worked toy example uses logistic regression as the primary model with a 90/10 train/validation split, explicitly built to monitor overfitting, and recommends selecting a decision threshold using an ROC curve to target high recall on the primary signal before the meta-model filters down to high precision. This directly validates the questionnaire's chosen architecture (rule-based primary + ML filter, not "replace the strategy with an ML black box"), and one commenter succinctly explains why this specific design is robust rather than merely convenient: "you think that for example moving average crossover works but you don't know when — you filter the bad trades to teach the algo where moving average worked... you ask the machine to identify the environment of when to invest and when not". This is a precise restatement of the questionnaire's own framing that the meta-model's role is trade filtering, not strategy replacement.[^2][^3][^4][^6]

## 2. Probability calibration

A classifier's raw output score is frequently not a well-calibrated probability — a model predicting "70%" confidence may not actually be correct 70% of the time — which matters directly for any downstream sizing decision (e.g., Kelly-based sizing, per Report 5) that depends on treating the model's output as a genuine probability rather than an arbitrary confidence score. **Platt scaling** fits a logistic regression model on the base classifier's raw scores against true labels, using a held-out validation set distinct from the training data to avoid leakage, and works well when the underlying miscalibration follows a roughly sigmoid shape. **Isotonic regression** is a non-parametric alternative that fits a monotonic step function rather than assuming any particular functional form, offering more flexibility but requiring more calibration data to avoid overfitting the calibration curve itself. The standard evaluation tool is the reliability diagram: bin predictions (commonly into ten bins), plot each bin's mean predicted probability against the true fraction of positive cases in that bin, and a well-calibrated model produces points falling near the diagonal.[^7][^8][^9][^10][^11]

One important practical caution directly relevant to the questionnaire's design: "there's no point in calibrating if the classifier is already good in this respect — first make a reliability diagram and if it looks like it could be improved, then calibrate". This argues that calibration should be a diagnosed-and-applied step, not a default pipeline stage applied unconditionally to every meta-model regardless of whether its raw outputs already happen to be well-calibrated. Scikit-learn's implementation recommends 5-fold cross-validation specifically for the calibration step itself, to avoid the calibration model overfitting to the same data used to fit it. Given the account's meta-model will need well-calibrated probabilities to feed sizing decisions (per E.208's design), this should be a required, explicitly tested step before any meta-model probability output is trusted for position sizing — untested calibration is a plausible, easy-to-miss source of systematic sizing error.[^8][^11][^7]

## 3. Model family selection: gradient boosting vs. logistic regression

Gradient boosting builds an ensemble of decision trees sequentially, where each new tree is trained to correct the residual errors of the prior ensemble, with a regularization/learning-rate parameter controlling how much each new tree is allowed to influence the combined prediction — this sequential error-correction process allows the model to capture complex, nonlinear feature interactions that a single linear model cannot. Logistic regression, by contrast, is described as "a foundational model for binary classification" that estimates probability via a linear combination of features, offering interpretability and resistance to overfitting on small samples as its principal advantages over more complex alternatives. Academic work combining the two directly — cascading logistic regression onto gradient-boosted decision trees for stock-index forecasting — found the hybrid architecture improved on either model used alone, suggesting the two approaches are complementary rather than strictly competing choices.[^12][^13][^14]

A comparative empirical study testing Logistic Regression, Random Forest, XGBoost, and Support Vector Machines on momentum-based equity signals concluded plainly that "the models, in their current form, are too simplistic to achieve sustained efficiency in real-world trading conditions" — a direct, sobering caution against assuming a more sophisticated model family automatically produces a tradeable edge simply by virtue of algorithmic complexity. Given the questionnaire's small-sample-size constraint (a v1 meta-model will have far fewer labeled trades than any of the academic studies reviewed here), the practical recommendation is to start with logistic regression or a shallow gradient-boosted tree with strong regularization, treating model complexity as something to earn through demonstrated out-of-sample improvement (via the walk-forward and DSR/PBO framework from Report 4) rather than assuming a more complex model family is default-superior.[^14]

## 4. Concept drift and retraining governance

Concept drift refers to a change over time in the actual statistical relationship between features and the target outcome — as distinct from **data drift** (a change in the distribution of the input features themselves, without necessarily changing the feature-target relationship). This distinction matters directly for the questionnaire's meta-model governance: a shift in average trade volatility might be data drift (the inputs look different, but the same features-to-outcome relationship may still hold), whereas a shift in which features actually predict trade success following a genuine regime change is concept drift proper, and the two may require different responses.[^15][^16][^17]

Formal drift-detection algorithms cited across sources include the Page-Hinkley test and the Kolmogorov-Smirnov test for detecting distributional shifts in incoming data, and ADWIN (adaptive windowing) specifically for detecting concept drift by comparing statistics across adaptive sub-windows of a data stream. A simpler, more universally applicable approach requiring no specialized drift-detection algorithm at all is direct performance monitoring: "regularly evaluating the performance of the ML model on a holdout dataset or in production can help to identify any decline in accuracy... that may indicate model drift" — this is functionally the same principle as the questionnaire's own N.264 CUSUM-based edge-decay monitoring from Report 6, applied specifically to the meta-model's own predictive accuracy rather than the underlying strategy's raw P&L.[^16]

Three governance models for responding to detected drift are described: **scheduled retraining** (periodic, regardless of detected drift), **threshold-triggered retraining** (only when monitored performance drops below a defined threshold), and **continuous/passive retraining** (retraining on every new data batch, appropriate specifically for gradual rather than abrupt drift). Given the account's own monthly full-strategy-review cadence (N.269) and the DSR/PBO honest-trial-recording requirement from Report 4, a **threshold-triggered** approach is the best fit: it avoids the "silent overfitting via repeated retraining" risk of continuous retraining while still responding faster than a purely calendar-based schedule if the meta-model's live accuracy degrades meaningfully between scheduled reviews. Critically, one source's caution about retraining decisions applies directly here: any retrain event, and the reasoning behind it, must be logged as part of the same experiment registry Report 4 identified as mandatory infrastructure — an undocumented retrain is functionally equivalent to an undocumented parameter search trial, and defeats the same honest-trial-counting requirement underlying DSR/PBO.[^17][^15][^16]

## 5. Explainability and feature attribution

SHAP (SHapley Additive exPlanations) is the dominant model-agnostic explainability method across the sources reviewed, using Shapley values from cooperative game theory to fairly distribute a prediction's total deviation from baseline across each contributing input feature, producing both local (per-prediction) and global (aggregate feature-importance) explanations. Its stated practical uses directly relevant to a live trading meta-model include model debugging during error analysis, detecting which specific features are driving degraded predictions when performance declines (directly useful for diagnosing whether a drift event is concentrated in one or two specific features versus a broad, system-wide shift), and general fairness/consistency auditing of model behavior.[^18][^19][^20][^21]

For this account's specific context, SHAP serves two concrete governance purposes beyond generic model interpretability: first, it gives a mechanism to verify a meta-model is not secretly relying on a feature that violates point-in-time integrity (a lookahead-bias check applied at the feature-attribution level, complementing Report 4's dataset-level lookahead checks); second, per the covariate-drift mitigation workflow described in the drift literature, SHAP-based feature importance allows a specific, low-impact drifting feature to be identified and removed from the model rather than requiring a full retrain in response to every detected drift event. This gives the meta-model governance process a graduated response option (remove one drifting low-importance feature) sitting between "do nothing" and "full retrain," which is a more surgical and auditable response than either extreme.[^22]

## 6. Does ML reliably beat simple rules in trading?

This is the most consequential finding in this report, and it should temper any expectation that adding ML sophistication will automatically or reliably improve on tradekit's existing rule-based approach. A rigorous academic study built a realistic forex day-trading market simulator and tested four distinct genres of machine learning algorithms over ten years of simulated conditions, explicitly incorporating realistic transaction costs and market frictions; the result was that "most ML algorithms perform as well as random," with only one algorithm (a specialized ensemble-selection approach) performing on par with a theoretical minimax benchmark. The paper's own stated explanation for this result is directly relevant to this account's design: "it is extremely difficult to make accurate predictions of the future in a noisy zero-sum market that changes over time," trades incur costs regardless of outcome, and individual/small traders lack the resources and market power that larger institutional participants possess. A separate industry-commentary source echoes this directly: "the appeal of machine learning is that it can extract patterns too complex or too subtle for hand-coded rules to capture. The challenge is that" the same complexity that lets ML fit subtle patterns also makes it far easier to fit noise that will not persist out-of-sample.[^23][^24]

This finding does not undermine the meta-labeling architecture specifically — meta-labeling's entire design philosophy (filter, don't predict; ML judges *conditions* around an already-identified opportunity rather than discovering new opportunities from raw price data) is structurally different from, and considerably more modest in its ambition than, the direction-prediction approach that the cited forex study and most failed retail ML strategies attempt. The practical implication for this account is that ML should be scoped exactly as the questionnaire already scopes it — a conditional trade filter layered on top of a rule-based primary signal, evaluated with the full rigor of Report 4's validation framework — rather than expanded into a more ambitious direction-prediction role, where the evidence base for ML's edge is considerably weaker and the failure literature considerably more consistent and cautionary.

## Synthesis for the ML architecture

The evidence strongly supports the meta-labeling filter design already specified in the questionnaire (E.207–208) as the correctly scoped and best-evidenced application of ML to this system; recommends adding an explicit, tested calibration step (Platt scaling or isotonic regression, chosen based on a reliability-diagram diagnosis rather than applied by default) before the meta-model's probability output feeds any sizing decision; supports starting with the simplest defensible model family (logistic regression or a shallow, heavily regularized gradient-boosted tree) given the account's necessarily small labeled-trade sample, with complexity earned through demonstrated walk-forward improvement rather than assumed; recommends threshold-triggered retraining governance logged into the same experiment registry Report 4 requires for honest trial-counting; and recommends adopting SHAP-based feature attribution both as a lookahead-bias audit tool and as a graduated, surgical response option to detected drift. Most importantly, the direction-prediction failure literature should be read as a firm boundary the ML component must not cross — the moment a "meta-model" starts effectively generating its own directional opportunities rather than filtering an existing rule-based signal, it inherits the far weaker evidence base and well-documented failure pattern this section describes.
